# Vue 2 BMP 页面开发工作流

这是一个工作流编排技能，按照固定的五步流程依次调用其他 skill 来完成 Vue 2 + Element UI 业务页面的开发与质量保障。每一步的输出摘要会作为下一步的输入上下文，且每一步执行前都需要用户确认。

## 为什么需要这个工作流

单独生成一个页面后，往往还需要代码审查、错误排查、性能优化、注释文档编写才能交付高质量代码。手动依次调用多个 skill 既繁琐又容易遗漏步骤，且各步骤之间的上下文无法自动传递。本工作流将这些步骤编排为一个自动化的流水线，每步之间衔接有序、上下文连贯。

## 工作流步骤

整个工作流包含五个阶段，严格按顺序执行：

| 阶段 | Skill | 目的 | 输入 | 输出 |
|------|-------|------|------|------|
| 1 | vue2-elementui-page-creator | 生成完整的 CRUD 表格页面 | 用户的需求描述 | 生成的文件列表 + 代码摘要 |
| 2 | frontend-codereview | 审查生成代码的规范、质量、安全性 | 阶段 1 的代码摘要 + 文件路径 | 代码审查报告摘要 |
| 3 | frontend-error-inspect | 排查潜在的运行时错误和布局问题 | 阶段 2 的审查报告摘要 + 文件路径 | 错误排查报告摘要 |
| 4 | frontend-performance-inspect | 审查性能瓶颈和 Web Vitals | 阶段 3 的排查报告摘要 + 文件路径 | 性能优化报告摘要 |
| 5 | frontend-doc-creator | 为生成的代码编写 JSDoc/TSDoc 注释文档 | 阶段 4 的性能审查摘要 + 文件路径 | 注释文档生成报告摘要 |

## 执行流程

### 阶段 0：收集需求

在开始工作流之前，先向用户收集页面开发需求：

- **页面名称**：什么实体？（如 "prisoner"、"device"、"visitor"）
- **表格列**：需要显示哪些列？每列的 label、prop、类型（text/tag/date/image 等）
- **搜索字段**：哪些字段需要搜索？（input/select/date-picker）
- **CRUD 操作**：需要添加？编辑？删除？其他操作？
- **API 路径**：后端接口前缀（如 `/basic/api/prisoner`）
- **权限标识**：如果使用 `v-permission`，权限 key 是什么？

如果用户提供了大致描述，推断合理的默认值而不是逐项追问。

收集完需求后，向用户展示完整的工作流计划：

```
📋 工作流计划：

1️⃣ 页面生成（vue2-elementui-page-creator）
   → 根据需求生成 index.vue、addXxx.vue、editXxx.vue、API 文件

2️⃣ 代码审查（frontend-codereview）
   → 对生成的代码进行规范、质量、安全性审查

3️⃣ 错误排查（frontend-error-inspect）
   → 排查潜在的运行时错误和布局问题

4️⃣ 性能审查（frontend-performance-inspect）
   → 审查性能瓶颈，给出 Web Vitals 优化建议

5️⃣ 注释文档编写（frontend-doc-creator）
   → 为生成的代码函数编写 JSDoc/TSDoc 注释文档

确认后将从阶段 1 开始，每一步执行前都会询问你是否继续。
```

### 阶段 1：页面生成

调用 `vue2-elementui-page-creator` skill：

1. **向用户确认**：展示阶段 1 的具体执行内容，询问"是否开始阶段 1：页面生成？"
2. **调用 skill**：使用 `use_skill` 工具调用 `vue2-elementui-page-creator`
3. **执行生成**：按照该 skill 的指引，收集需求并生成完整页面文件
4. **提取摘要**：生成完成后，提取以下信息作为阶段 2 的输入：

```
📝 阶段 1 摘要：
- 生成文件：[文件路径列表]
- 页面结构：[搜索字段列表] + [表格列列表] + [CRUD 操作]
- API 接口：[列出生成的 API 函数名和路径]
- 关键代码特征：[使用的组件模式、数据结构等]
```

5. **展示摘要**：将摘要展示给用户，然后进入阶段 2 的确认环节

### 阶段 2：代码审查

调用 `frontend-codereview` skill：

1. **向用户确认**：展示阶段 2 的具体执行内容和阶段 1 的摘要，询问"是否开始阶段 2：代码审查？"
2. **调用 skill**：使用 `use_skill` 工具调用 `frontend-codereview`
3. **执行审查**：按照该 skill 的指引，对阶段 1 生成的所有文件进行审查
4. **传递上下文**：在审查时，告知审查 skill 以下上下文信息：
   - 这些文件是刚通过 vue2-elementui-page-creator 生成的
   - 文件路径列表
   - 页面的业务功能和结构特征（来自阶段 1 摘要）
5. **提取摘要**：审查完成后，提取以下信息作为阶段 3 的输入：

```
📝 阶段 2 摘要：
- 审查文件：[文件路径列表]
- 严重问题：[数量] — [简要列出每个严重问题]
- 建议优化：[数量] — [简要列出关键建议]
- 亮点：[简要列出]
- 需重点关注的代码区域：[文件:行号列表]
```

6. **展示摘要**：将摘要展示给用户。如果存在严重问题，询问用户是否需要先修复后再继续，如果用户选择修复，则暂停工作流协助修复，修复后重新执行阶段 2 审查

### 阶段 3：错误排查

调用 `frontend-error-inspect` skill：

1. **向用户确认**：展示阶段 3 的具体执行内容和阶段 2 的摘要，询问"是否开始阶段 3：错误排查？"
2. **调用 skill**：使用 `use_skill` 工具调用 `frontend-error-inspect`
3. **执行排查**：按照该 skill 的指引，对生成的代码进行错误排查
4. **传递上下文**：在排查时，告知排查 skill 以下上下文信息：
   - 代码审查阶段发现的严重问题和建议（来自阶段 2 摘要）
   - 需重点关注的代码区域
   - 文件路径列表
5. **提取摘要**：排查完成后，提取以下信息作为阶段 4 的输入：

```
📝 阶段 3 摘要：
- 排查范围：[文件路径列表]
- 发现的错误风险：[数量] — [简要列出每个错误风险]
- 布局问题：[如有，简要列出]
- Vue 2 + Element UI 专项问题：[如有，简要列出]
- 建议修复的优先级排序：[从高到低]
```

6. **展示摘要**：将摘要展示给用户。如果存在高风险错误，询问用户是否需要先修复后再继续

### 阶段 4：性能审查

调用 `frontend-performance-inspect` skill：

1. **向用户确认**：展示阶段 4 的具体执行内容和阶段 3 的摘要，询问"是否开始阶段 4：性能审查？"
2. **调用 skill**：使用 `use_skill` 工具调用 `frontend-performance-inspect`
3. **执行审查**：按照该 skill 的指引，对生成的代码进行性能审查
4. **传递上下文**：在审查时，告知性能审查 skill 以下上下文信息：
   - 代码审查和错误排查阶段发现的问题（来自阶段 2、3 摘要）
   - 特别关注这些可能影响性能的代码区域
   - 文件路径列表
5. **提取摘要**：审查完成后，输出摘要作为阶段 5 的输入：

```
📝 阶段 4 摘要：
- 审查范围：[文件路径列表]
- 性能问题：[数量] — [简要列出每个性能问题]
- Web Vitals 影响：[LCP/FID/CLS/INP/TTFB 的预估状态]
- 优化建议优先级：[从高到低]
```

6. **展示摘要**：将摘要展示给用户，然后进入阶段 5 的确认环节

### 阶段 5：注释文档编写

调用 `frontend-doc-creator` skill：

1. **向用户确认**：展示阶段 5 的具体执行内容和阶段 4 的摘要，询问"是否开始阶段 5：注释文档编写？"
2. **调用 skill**：使用 `use_skill` 工具调用 `frontend-doc-creator`
3. **执行生成**：按照该 skill 的指引，为阶段 1 生成的所有文件的函数编写 JSDoc/TSDoc 注释
4. **传递上下文**：在编写注释时，告知注释文档 skill 以下上下文信息：
   - 前四个阶段发现的代码结构和业务功能特征（来自阶段 1 摘要）
   - 代码审查阶段指出的命名和规范问题（来自阶段 2 摘要）
   - 文件路径列表
5. **提取摘要**：注释生成完成后，输出最终摘要：

```
📝 阶段 5 摘要：
- 处理文件：[文件路径列表]
- 添加注释函数数：[数量]
- 跳过已有注释函数数：[数量]
- 注释风格：[JSDoc / TSDoc]
- 无法确定用途的函数：[如有，列出]
```

### 最终汇总

五个阶段全部完成后，输出完整的工作流汇总报告：

```markdown
# 🏁 BMP 页面开发工作流 — 完成报告

## 页面信息
- 页面名称：xxx
- 生成文件：[文件列表]

## 各阶段结果

| 阶段 | 结果 | 关键发现 |
|------|------|----------|
| 页面生成 | ✅ 完成 | [关键特征] |
| 代码审查 | ✅ 完成 | [严重问题数 + 建议数] |
| 错误排查 | ✅ 完成 | [错误风险数] |
| 性能审查 | ✅ 完成 | [性能问题数] |
| 注释文档 | ✅ 完成 | [添加注释函数数] |

## 综合建议

按优先级汇总所有阶段发现的问题和优化建议：
1. [最高优先级 — 来自哪个阶段]
2. [次高优先级]
3. ...

## 下一步

- [如有需要修复的问题，建议逐一修复]
- [如所有问题已解决，建议提交代码]
```

## 关键规则

- **严格顺序**：五个阶段必须按 1→2→3→4→5 的顺序执行，不可跳过或颠倒
- **用户确认**：每个阶段开始前必须询问用户是否继续执行。用户可以选择：
  - "继续" — 执行当前阶段
  - "跳过" — 跳过当前阶段，但需确认是否继续后续阶段
  - "暂停" — 暂停工作流，等待用户回来后继续
  - "终止" — 终止整个工作流，输出当前已完成的阶段摘要
- **上下文传递**：每个阶段必须将上一阶段的摘要作为上下文信息传递给当前阶段的 skill，确保后续步骤能够聚焦于前面发现的问题
- **摘要格式**：每个阶段的摘要必须包含文件路径、关键发现数量和具体内容，确保信息充分且简洁
- **修复中断**：如果某阶段发现严重问题，应询问用户是否需要先修复。修复后可根据用户选择重新执行该阶段或继续下一阶段
